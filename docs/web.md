# 웹 전환 검토 정리

## 목적
- 현재 Electron 기반 데스크톱 앱을 웹으로 전환할 수 있을지 검토한 내용을 정리한다.
- 이 문서는 새 웹용 프로젝트를 만들고 다음 세션으로 인계하기 위한 요약 문서다.

## 결론 요약
- `전체를 바로 웹 + Vercel 무료`로 옮기는 것은 리스크가 크다.
- 대신 현재 프로젝트는 보존하고, 별도 웹 프로젝트를 만들어 점진적으로 옮기는 것이 안전하다.
- 웹으로 옮기기 쉬운 기능과 어려운 기능이 명확히 갈린다.
- 협정보드 저장/불러오기, 메일 발송, 엑셀 파일 생성은 웹에서도 가능하다.
- 반면 로컬 Excel 자동 제어, Electron IPC, 로컬 파일 경로 중심 워크플로우는 그대로 옮기기 어렵다.

## 현재 프로젝트 특성
- 단순 React SPA가 아니라 Electron 메인 프로세스 + preload + IPC 구조를 사용 중이다.
- 로컬 Excel 파일을 직접 읽고 가공한다.
- 일부 기능은 Windows PowerShell을 통해 실행 중인 Excel 프로그램을 직접 제어한다.
- 로컬 JSON 저장, 로컬 DB, 첨부파일 경로, 메일/SMPP 등 데스크톱 성격이 강하다.

관련 파일 예시:
- `main.js`
- `preload.js`
- `searchLogic.js`
- `src/main/features/excel/excelAutomation.js`
- `src/main/features/records/recordsDatabase.js`
- `src/main/features/mail/smtpService.js`
- `src/main/features/smpp/smppService.js`

## 웹 전환 적합도

### 웹으로 옮기기 쉬운 영역
- 업체 검색 UI
- 계산 로직
- 협정보드 화면 자체
- 협정 저장/불러오기
- 메일 발송
- 엑셀 템플릿 기반 파일 생성 및 다운로드

### 웹으로 옮기기 어려운 영역
- Electron IPC에 의존한 기능
- 로컬 폴더 경로를 직접 선택하고 계속 사용하는 흐름
- 사용자 PC에서 이미 열려 있는 Excel 프로그램을 직접 제어하는 기능
- Windows 전용 PowerShell/COM 자동화
- 로컬 첨부파일 경로를 그대로 저장하는 구조

## Vercel 무료 배포에 대한 판단
- 프론트 배포 및 가벼운 API 제공 용도로는 적합하다.
- 협정보드 저장/불러오기 같은 일반 CRUD는 무료 티어로도 시작 가능성이 높다.
- 다만 대량 메일, 대용량 파일 처리, 첨부파일 누적, 무거운 배치 작업까지 같이 얹으면 무료 티어가 빨리 답답해질 수 있다.

공식 문서 확인 기준:
- Vercel Functions Hobby는 메모리/실행시간 제한이 있다.
- Vercel Blob은 all plans에서 사용 가능하다.
- 장기 운영 시에는 DB/스토리지 사용량을 별도로 봐야 한다.

참고 링크:
- https://vercel.com/docs/functions/limitations
- https://vercel.com/docs/storage
- https://vercel.com/docs/vercel-blob

## 메일 기능 웹 전환 판단
- 구현 가능하다.
- 현재 `nodemailer` 기반 SMTP 발송은 서버 함수로 옮길 수 있다.
- 단, SMTP 계정/비밀번호를 브라우저에 두면 안 되고 서버 환경변수로 옮겨야 한다.
- 소량 테스트 메일은 가능하지만 운영형 대량 발송은 메일 API 서비스 사용이 더 적합하다.

권장:
- 초기: SMTP 또는 간단한 메일 API
- 운영: Resend / SendGrid / Postmark 검토

## 엑셀 기능 웹 전환 판단

### 가능한 것
- 템플릿 `.xlsx` 파일에 데이터 채워서 결과 파일 생성
- 사용자가 결과 파일을 다운로드
- 업로드한 엑셀 파일을 읽고 가공한 뒤 새 파일로 반환

### 어려운 것
- 사용자의 PC에서 이미 열려 있는 Excel 프로그램에 직접 붙어서 셀을 읽고 쓰는 기능

정리:
- `엑셀 내보내기`는 웹에서도 가능
- `로컬 Excel 자동 제어`는 웹에서 사실상 불가에 가깝고 별도 데스크톱 helper가 필요할 수 있음

## 템플릿 저장 방식
Electron에서는 템플릿을 앱 설치본에 포함해 배포했지만, 웹에서는 사용자 로컬과 분리된다.

웹에서 가능한 템플릿 저장 방식:

### 1. 코드 저장소에 포함
- 프로젝트 내부 `templates/*.xlsx` 형태로 포함
- 배포 시 서버가 해당 템플릿을 읽어 엑셀 생성
- 가장 단순하고 초기 구축에 적합
- 템플릿 변경 시 재배포 필요

### 2. 오브젝트 스토리지 사용
- Vercel Blob / S3 / Supabase Storage 등에 저장
- 운영 중 템플릿 교체가 쉬움
- 구조는 조금 더 복잡해짐

현재 상황에서의 권장:
- 초기는 `repo 포함 방식`
- 나중에 운영 편의가 필요하면 `Blob/S3`로 분리

## 협정보드 저장/불러오기 웹 전환 판단
- 웹으로 바꾸면 현재처럼 사용자 PC 로컬에 JSON 파일을 저장하는 구조를 그대로 쓰기 어렵다.
- 저장한 협정 데이터는 서버 쪽 저장소에 두어야 한다.

중요:
- 꼭 JSON 파일 자체로 저장할 필요는 없다.
- 웹에서는 DB 저장이 더 일반적이고 관리가 쉽다.

### 선택지 1. JSON 파일 그대로 저장
- Blob/S3에 JSON 파일 단위로 저장
- 현재 구조와 개념은 비슷함
- 목록 조회, 검색, 수정, 사용자별 정리가 불편함

### 선택지 2. DB에 저장
- 협정 1건을 레코드 1건으로 저장
- 본문은 JSON 컬럼으로 저장하거나 일부 필드를 컬럼으로 분리
- 목록, 검색, 정렬, 자동저장, 공유, 권한 처리에 유리

권장:
- 협정보드 저장/불러오기는 DB 기반으로 재설계

## 무료 티어로 협정보드 저장/불러오기 가능 여부
- 가능성이 높다.
- 내부 툴 수준, 소규모 사용자, 협정 수가 많지 않은 상태라면 무료 티어로 시작 가능하다.
- 보통 구성은 `Vercel Hobby + 외부 DB 무료 티어` 조합이 현실적이다.

단, 아래 조건이 붙으면 유료 전환 가능성이 높아진다.
- 사용자 수 증가
- 데이터 누적 증가
- 첨부파일 저장
- 공유/권한/버전관리 고도화
- 대량 메일 또는 무거운 파일 처리

## 웹 전환 진행 방식 권장안
- 현재 Electron 프로젝트는 유지한다.
- 별도 웹 프로젝트를 새로 만들어 거기서 진행한다.
- 이유는 단순 마이그레이션이 아니라 제품 구조가 크게 달라지기 때문이다.
- 기존 프로젝트 안에서 바로 바꾸면 운영 중인 데스크톱 앱을 깨뜨릴 위험이 크다.

권장 순서:
1. 현재 프로젝트는 보존
2. 새 폴더 또는 새 저장소로 웹 프로젝트 시작
3. 공통 계산 로직만 우선 추출
4. 협정보드 저장/불러오기 웹 MVP 구현
5. 엑셀 내보내기 구현
6. 메일 기능 구현
7. 마지막에 필요 기능만 단계적으로 확장

## 웹 프로젝트에서 재사용/대체 대상

### 우선 재사용 검토
- `src/shared/**` 계산 로직
- 협정보드 관련 계산/규칙 로직
- 화면 흐름 중 프론트 UI 일부

### 대체 필요
- `main.js`
- `preload.js`
- `src/main/**`
- 로컬 파일 저장 로직
- Excel COM / PowerShell 자동화

## 새 웹 프로젝트 초기 권장 아키텍처
- 프론트: React
- 배포/API: Vercel
- 협정 저장: Neon 또는 Supabase 같은 Postgres 계열 무료 티어 검토
- 템플릿 저장: 초기에는 repo 포함, 필요 시 Blob/S3
- 메일 발송: 서버 함수 + 메일 API

## 한 줄 결론
- 원본 프로젝트는 유지하고, 새 웹 프로젝트에서 협정보드 중심으로 MVP를 새로 만드는 방향이 가장 안전하다.
