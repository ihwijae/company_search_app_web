# 노트북 서버 검토 및 현재 테스트 정리

## 1. 이 문서의 목적
- 다음 세션에서 `노트북을 서버처럼 사용할 수 있는지` 이어서 검토하기 위한 인계 문서
- 지금까지 나눈 대화, 현재 판단, 실제 테스트 결과를 한 번에 정리

## 2. 현재까지 정리된 핵심 판단

### 웹 전환 자체의 의미
- 웹 전환은 여전히 의미가 있음
- 이유:
  - 설치형 배포보다 중앙 배포가 쉬움
  - 여러 사용자가 같은 주소로 접근 가능
  - 코드 수정 후 반영이 Electron 패키징보다 단순함
  - 장기적으로 자동화 기능을 `UI`와 `실행 엔진`으로 분리하기 좋음

### 다만 서버 운영은 별도 문제
- 웹 전환과 `직접 서버 운영`은 다른 문제임
- 웹으로 바뀐다고 해서 인프라 부담이 자동으로 사라지지는 않음
- 노트북/개인 PC를 서버로 쓰면 아래를 직접 챙겨야 함
  - 실행 환경
  - 포트 개방/방화벽
  - WSL 포트 전달
  - 자동 재시작
  - HTTPS/외부 공개 방식
  - 백업

## 3. Vercel 관련 판단
- `Vercel 무료 플랜 자체가 전부 unusable` 하다고 보긴 어려움
- 현재 프로젝트에서 더 큰 병목은 `Vercel Blob 저장 구조` 쪽으로 판단
- 특히 `records/index.json` 전체 저장 방식은 작은 수정에도 Blob 고급 작업 사용량을 많이 먹을 수 있음
- 따라서 핵심은 `Vercel을 버릴지`보다 `Blob 의존 구조를 줄일지`에 가까움

### 대화 중 확인한 점
- Vercel Hobby 함수 메모리는 `2GB`
- 이 수치 자체보다 중요한 것은:
  - 큰 파일 처리
  - 대용량 JSON 전체 저장
  - 브라우저 자동화 같은 무거운 작업
- 즉 현재 우선 병목은 RAM보다 저장 구조/운영 구조일 가능성이 큼

## 4. 자동화 기능(로그인, 공인인증서 인증, 협정서 송신) 관련 판단
- Selenium/Playwright 같은 브라우저 자동화는 원래 무거운 작업임
- 무거워지는 건 웹앱 UI라기보다 `자동화 실행 환경`임
- 따라서 장기적으로는 아래처럼 분리하는 게 맞다는 판단

```text
웹앱(UI/API 일부)
  -> 작업 요청
  -> 상태 조회

자동화 워커
  -> Selenium / Playwright 실행
  -> 로그인 / 인증 / 전송
  -> 로그 / 재시도 / 실패 처리
```

### 핵심 결론
- `웹으로 가야 한다`의 이유가 단순히 `테스트 도구가 무거워서`는 아님
- 더 정확히는:
  - 배포와 업데이트를 쉽게 만들고
  - 무거운 자동화 작업을 사용자 앱과 분리해
  - 서버 측에서 중앙 관리하기 쉬워지기 때문

## 5. 노트북을 서버로 쓰는 방향에 대한 판단

### 가능 여부
- 가능은 함
- 특히 내부 테스트/소규모 운영에는 현실적인 선택지

### 장점
- 직접 서버 환경을 통제 가능
- 자동화 기능까지 붙이기 쉬움
- 장기적으로 중앙화된 구조를 만들 수 있음

### 단점
- 초기 세팅 필요
- 상시 실행, 절전 해제, 재시작 대응 필요
- 외부 공개가 필요하면 HTTPS/터널/도메인 문제를 해결해야 함
- 코드 수정 시 노트북에도 반영 절차가 필요함

### 코드 반영 관련 판단
- 노트북을 서버로 쓰면 개발 PC에서 수정 후 노트북에도 반영해야 함
- 즉:
  - `git push`
  - 노트북에서 `git pull`
  - `npm install` 필요시 수행
  - `npm run build`
  - 서버 재시작
- 장기적으로는 배포 스크립트 또는 자동화가 필요할 수 있음

## 6. 개인 PC를 임시 서버로 쓰는 방향
- 현재 단계에서는 `내가 쓰는 PC를 임시 서버로 쓰는 것`도 충분히 현실적이라는 판단
- 이유:
  - 이미 개발 환경이 있음
  - 수정 후 바로 반영 가능
  - 지금 접속 테스트도 성공함
- 다만 장기 운영에는 아래 리스크가 있음
  - 작업 중 PC 재부팅/절전/업데이트
  - 개발 작업과 운영 서버가 같은 머신에 섞임
  - 자동화 기능이 붙으면 자원 충돌 가능성 증가

## 7. 내부망 / 외부 공개 관련 판단

### 내부망 주소는 외부 공개 주소가 아님
- `192.168.x.x`, `10.x.x.x`, `172.16~31.x.x` 대역은 사설 IP
- 이런 주소는 내부 네트워크 안에서만 접근 가능
- 따라서 `http://192.168...` 주소로는 외부 누구나 접속하는 구조를 만들 수 없음

### 외부 공개가 필요하면 선택지는 사실상 2개
1. 포트포워딩 + 도메인 + HTTPS
2. 터널 서비스 사용

### 현재 대화 기준 판단
- 포트포워딩은 담당자 협조/이해가 애매한 상태
- 노트북은 Wi-Fi로 운영할 가능성이 있음
- 외부 누구나 접속 가능한 구조를 원함
- 따라서 장기적으로는 아래 중 하나가 더 현실적임
  - `클라우드 호스팅 + 외부 저장소`
  - `노트북 서버 + 터널 서비스`

### 터널 서비스 관련 판단
- `Cloudflare Tunnel`, `ngrok`, `Tailscale Funnel` 같은 방식은 가능
- 장점:
  - 포트포워딩 불필요
  - IP 변경 영향을 덜 받음
  - 외부 공개 URL 생성 가능
- 단점:
  - 노트북이 항상 켜져 있어야 함
  - 터널 프로세스도 살아 있어야 함
  - 사내 네트워크 정책 영향을 받을 수 있음

## 8. WSL 네트워크 관련 실제 테스트 결과

### 테스트 목적
- 현재 내 PC에서 WSL로 띄운 웹 서버가 같은 사내망 다른 자리에서도 접속 가능한지 확인

### 테스트 중 확인한 점
- WSL 서버를 실행하면 터미널에 아래처럼 내부/네트워크 주소가 표시됨
- 하지만 `WSL 내부 주소(예: 172.22.x.x)`는 다른 자리에서 직접 접근하는 주소로 쓰기 어려움
- 다른 자리에서 접속할 때는 `윈도우 호스트 IP(예: 192.168.x.x)`를 써야 함

### 실제로 발생한 문제
- `localhost`로는 접속되는데
- 같은 PC에서 `http://윈도우IP:포트` 로는 처음에 접속되지 않았음

### 원인 판단
- WSL 서버가 외부 LAN 요청을 직접 받지 못했고
- 윈도우가 `LAN 요청 -> WSL 포트` 전달을 자동으로 제대로 해주지 않았던 상태

### 해결 방법
- 윈도우 방화벽에서 해당 포트 허용
- `netsh interface portproxy` 로 `윈도우 포트 -> WSL 포트` 전달 설정

### 결과
- 내 자리에서 `윈도우 IP:포트` 접속 성공
- 다른 자리 PC에서도 같은 주소로 접속 성공

### 요약 결론
- `WSL에서 서버를 띄워도`, 같은 사내망 다른 기기에서는 보통 `윈도우 IP` 로 접속해야 함
- 단, 그 전에 `portproxy` 와 방화벽 설정이 필요할 수 있음

## 9. Wi-Fi 환경 관련 실제 판단
- 처음에는 Wi-Fi 환경이라 접속이 안 되는 것처럼 보였음
- 그러나 실제로는 `Wi-Fi 자체가 불가능한 것`은 아니었음
- 적절한 포트 전달과 방화벽 설정 후에는 Wi-Fi 상태에서도 접속 가능함

### 다만 주의점
- 윈도우 IP는 네트워크 재연결/재부팅/DHCP 재할당 시 바뀔 수 있음
- WSL IP는 윈도우 IP보다 더 잘 바뀔 수 있음
- 특히 `portproxy` 는 WSL IP를 연결 대상으로 쓰므로, WSL IP가 바뀌면 다시 설정해야 할 수 있음

## 10. 모바일 접속 관련 판단
- 사내망 내부 주소로 열어둔 서버는 `사내망에 들어온 기기만` 접속 가능
- 모바일도 동일

### 가능한 경우
- 회사 Wi-Fi에 연결된 모바일
- 또는 VPN으로 사내망에 붙은 모바일

### 안 되는 경우
- LTE/5G 등 사내망 밖 모바일
- 회사 Wi-Fi 정책상 내부 PC 접근이 차단된 무선망

## 11. 직접 서버 운영 시 앞단 웹서버에 대한 판단
- 직접 서버를 운영하면 보통 `nginx` 또는 `caddy` 같은 앞단 웹서버를 둠
- 목적:
  - `80/443` 포트 처리
  - HTTPS 종료
  - 리버스 프록시
  - 정적 파일 서빙
  - 보안 헤더/압축 등 기본 웹서버 역할

### 판단
- `nginx`가 일반적인 선택이지만
- 세팅 단순성만 보면 `caddy`가 더 쉬울 수 있음
- 다만 지금 단계에서는 아직 여기에 들어가기 전 상태

## 12. 노트북 세팅은 다음 세션으로 미룸
- 현재 세션에서는 노트북 세팅을 시작하지 않음
- 다음 세션에서 진행할 예정

### 다음 세션에서 가장 먼저 할 일
1. 노트북에 `WSL` 설치
2. `Ubuntu` 초기 설정
3. `Node.js`, `npm`, `git` 설치
4. 프로젝트 클론 및 `npm install`
5. `npm run build`
6. `npm run preview -- --host 0.0.0.0` 또는 운영용 실행 방식 검토
7. 방화벽/포트 전달 설정
8. 같은 사내망 다른 PC에서 접속 확인

## 13. 다음 세션에서 결정해야 할 큰 방향

### 선택지 A. 노트북을 실제 서버처럼 운영
- 장점:
  - 자동화 기능까지 한곳에서 돌리기 쉬움
- 단점:
  - 운영 세팅 부담
  - 외부 공개 시 터널/HTTPS/보안 문제 처리 필요

### 선택지 B. 현재 PC를 임시 서버로 운영
- 장점:
  - 바로 테스트 가능
  - 개발 환경 이미 준비됨
- 단점:
  - 개발과 운영이 섞임

### 선택지 C. 클라우드 중심 구조 유지
- 예:
  - 프론트/웹앱: Vercel
  - 저장소: R2 등 외부 스토리지
  - 자동화: 별도 서버/워커
- 장점:
  - 배포/접속/HTTPS 단순
- 단점:
  - 자동화 워커 쪽은 별도 설계 필요

## 14. 현재 시점의 실무적 결론
- 웹 전환 방향 자체는 유지할 가치가 있음
- 다만 `노트북을 메인 서버로 둘지`, `클라우드 중심으로 갈지`는 아직 확정 단계는 아님
- 자동화 기능까지 고려하면 장기적으로는 아래 구조가 더 적합해 보임

```text
웹앱
  + 저장소
  + 자동화 워커(분리)
```

- 다음 세션에서는 우선 `노트북에서 WSL 기반으로 서버 실행이 가능한지`부터 검증하는 것이 적절함
