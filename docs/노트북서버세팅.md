# 노트북 서버 세팅 정리

## 1. 이 문서의 목적
- 노트북 WSL 환경을 실제 운영 서버처럼 사용하기 위해 진행한 설정 내용을 정리
- 다음 세션에서 운영 환경을 복구하거나 다른 머신으로 옮길 때 참고할 기준 문서
- 개발은 `데스크탑`, 운영은 `노트북`으로 분리한다는 현재 방식을 명확히 기록

## 2. 현재 운영 방향

### 역할 분리
- `데스크탑 PC`에서 개발을 진행한다.
- `노트북 WSL`은 운영 서버처럼 사용한다.

### 현재 판단
- 개발과 운영을 같은 머신에서 섞지 않는 것이 더 낫다.
- 비용 문제 때문에 당분간은 외부 Blob 같은 유료 저장소보다 `노트북 로컬 저장`을 우선 고려한다.
- 외부 공개는 `직접 포트포워딩` 대신 `Cloudflare Tunnel`을 사용한다.

## 3. 노트북 기본 환경

### OS / 실행 환경
- Windows 노트북에서 `WSL2 + Ubuntu` 사용
- 프로젝트는 WSL 내부 리눅스 경로에서 실행

### 기본 설치 흐름
```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y git curl unzip build-essential bzip2
```

```bash
curl -fsSL https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install --lts
node -v
npm -v
```

### 프로젝트 준비
```bash
mkdir -p ~/projects
cd ~/projects
git clone <repo-url>
cd company_search_app_web
npm install
npm run build
```

## 4. 사내망 접속 테스트 정리

### 기본 원칙
- WSL에서 서버를 띄워도 다른 PC는 보통 `WSL 내부 IP`가 아니라 `Windows 호스트 IP`로 접속해야 한다.
- 예: `http://192.168.x.x:4173`

### 실제 확인한 점
- 노트북 내부에서는 WSL 서버 접속 가능
- 다른 자리 PC에서도 접속 성공
- 문제처럼 보였던 일부 접속 실패는 `노트북 서버 문제`가 아니라 `접속하던 PC의 네트워크 경로 문제`였다.

### 중요한 트러블슈팅 결과
- 접속이 안 되던 PC는 `Wi-Fi`를 사용 중이었다.
- 같은 PC를 `유선 LAN`으로 연결하니 접속이 정상 동작했다.
- 즉 서버 문제가 아니라, `해당 Wi-Fi 경로에서는 내부 서버 접근이 제한`된 상태였다.

### 해석
- 같은 회사 네트워크처럼 보여도 `Wi-Fi`와 `유선`이 서로 다른 내부망/VLAN일 수 있다.
- 일부 무선망은 인터넷만 되고 내부 장치 접근이 막혀 있을 수 있다.
- 따라서 사내 테스트에서는 `유선 연결 기준`이 더 안정적이다.

## 5. 외부 공개 방식

### 왜 내부 IP 방식으로는 부족한가
- `192.168.x.x` 같은 주소는 사내망 내부에서만 접근 가능하다.
- 모바일 LTE/5G나 외부 네트워크에서는 접속할 수 없다.

### 선택한 방식
- 도메인: `jium.store`
- 외부 공개 주소: `app.jium.store`
- 공개 방식: `Cloudflare Tunnel`

### 이 방식을 선택한 이유
- 공유기 포트포워딩이 필요 없다.
- 공인 IP 또는 사설 IP를 직접 등록할 필요가 없다.
- 노트북 IP가 바뀌어도 `cloudflared`가 다시 Cloudflare에 연결하면 동일 도메인으로 접근 가능하다.
- HTTPS가 Cloudflare 구간에서 자동 처리된다.

## 6. Cloudflare Tunnel 실제 설정 내용

### 터널 생성
```bash
cloudflared tunnel create company-search
```

생성된 터널 ID:
```text
c90be071-466e-4812-acb9-23f509851e16
```

### DNS 연결
```bash
cloudflared tunnel route dns company-search app.jium.store
```

### 설정 파일
경로:
```text
/home/ihwijae/.cloudflared/config.yml
```

내용:
```yaml
tunnel: c90be071-466e-4812-acb9-23f509851e16
credentials-file: /home/ihwijae/.cloudflared/c90be071-466e-4812-acb9-23f509851e16.json

ingress:
  - hostname: app.jium.store
    service: http://localhost:4173
  - service: http_status:404
```

### 실행 명령
앱 서버:
```bash
cd /home/ihwijae/projects/company_search_app_web
npm run preview -- --host 0.0.0.0
```

터널:
```bash
cloudflared tunnel run company-search
```

### 현재 확인된 결과
- `https://app.jium.store` 접속 성공
- 외부 HTTPS 주소로 정상 접근 가능
- 노트북 내부 IP를 직접 등록하지 않아도 동작

## 7. Vite 설정에서 추가한 내용

Cloudflare Tunnel을 통해 들어오는 Host 헤더를 Vite preview가 허용하지 않아 아래 설정을 추가했다.

파일:
```text
vite.config.js
```

추가된 설정:
```js
preview: {
  port: 4173,
  strictPort: true,
  allowedHosts: ['app.jium.store'],
},
```

이 설정이 없으면 아래와 같은 에러가 발생했다.
```text
Blocked request. This host ("app.jium.store") is not allowed.
```

## 9. 파일 저장에 대한 현재 판단

### 현재 방향
- 비용 문제 때문에 당분간은 `노트북 로컬 디스크 저장`을 우선 고려한다.
- 서버가 WSL에서 돌아가므로 업로드 파일도 기본적으로 `WSL 내부 경로`에 저장된다.

### 권장 방향
- 프로젝트 코드와 업로드 데이터는 분리한다.
- 예:
```text
/home/ihwijae/projects/company_search_app_web
/home/ihwijae/app-data/company-search/uploads
```

### 주의점
- 노트북이 꺼지면 웹과 파일 저장 기능이 같이 중단된다.
- 백업 전략이 꼭 필요하다.
- 나중에 Blob/S3 같은 외부 저장소로 옮기기 쉽도록 저장 로직은 별도 계층으로 분리하는 것이 좋다.

## 10. 현재 운영 시 주의사항
- 노트북이 켜져 있어야 한다.
- WSL이 실행 중이어야 한다.
- 앱 서버가 떠 있어야 한다.
- `cloudflared tunnel run company-search` 가 실행 중이어야 한다.
- 노트북 IP가 바뀌는 것은 큰 문제가 아니지만, `cloudflared` 프로세스가 죽으면 외부 접속이 끊긴다.
- 회사 Wi-Fi에서는 내부 장치 접근이 제한될 수 있으므로, 내부 테스트는 유선 연결이 더 안전하다.
