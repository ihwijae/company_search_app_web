# 노트북 세션 인계

## 1. 이 문서의 목적
- 다음 세션에서 `노트북 WSL 환경`에서 바로 작업을 이어가기 위한 인계 문서
- 현재 결정 사항, 진행 상태, 막힌 지점, 다음 작업 순서를 한 번에 정리

## 2. 현재 결정된 운영 방향

### 역할 분리
- `현재 PC`는 개발 서버로 사용
- `노트북`은 실제 운영 서버처럼 사용

### 이 방향을 택한 이유
- 개발 중 실험, 재시작, 에러가 운영 환경에 바로 섞이지 않게 하기 위함
- 노트북을 비교적 안정적으로 켜 두고 사내망 접속 테스트를 하기 위함
- 이후 배포 절차를 `개발 -> Git 반영 -> 노트북 pull/build/restart` 구조로 단순화하기 위함

## 3. 현재 노트북 진행 상태
- `WSL2` 설치 완료
- 아직 다음 항목은 진행 전 또는 진행 중
  - Ubuntu 초기 설정
  - 필수 패키지 설치
  - Node.js / npm 설치
  - Codex 설치 여부
  - 프로젝트 정상 설치 및 실행 확인

## 4. 현재 세션에서 정리한 기본 작업 순서
1. WSL(Ubuntu) 실행
2. 패키지 목록 갱신 및 기본 업그레이드
3. 필수 도구 설치
4. `nvm` 설치
5. `nvm`으로 `Node.js LTS` 설치
6. 작업 폴더 생성
7. 저장소 clone
8. `.env`, `.env.local`, `db/`, `template/`, `템플릿/` 등 운영에 필요한 파일 복원 여부 확인
9. `npm install`
10. `npm run build`
11. `npm run preview -- --host 0.0.0.0`
12. 노트북 내부 접속 확인
13. 같은 사내망 다른 PC에서 접속 가능하도록 방화벽/포트 전달 설정 검토

## 5. 사용자가 헷갈렸던 부분 정리

### `apt upgrade -y`
- `-y`는 확인 질문에 자동으로 `yes`를 응답하는 옵션

### `curl ... | bash`
- 이 명령은 `nvm` 자체를 설치하는 명령

### `nvm install --lts`
- 위에서 설치한 `nvm`을 이용해 `Node.js`를 설치하는 명령
- 즉 `nvm 설치`와 `Node 설치`는 다른 단계

### `git clone`
- WSL 안에서 `현재 위치한 폴더` 기준으로 저장소가 생성됨
- 권장 경로는 WSL 리눅스 파일시스템 내부의 `~/projects`

### `mkdir -p ~/projects`
- 홈 디렉토리 아래에 `projects` 폴더를 만드는 명령
- 이미 폴더가 있어도 에러 없이 통과

### `npm install`
- `package.json`, `package-lock.json` 기준으로 필요한 Node 패키지를 자동 설치
- 다만 일부 패키지는 리눅스 시스템 패키지를 추가로 요구할 수 있음

## 6. 현재 실제로 발생한 설치 이슈

### 상황
- 노트북 WSL에서 프로젝트 clone 후 `npm install` 실행
- 설치 중 `phantomjs-prebuilt` 관련 오류 발생

### 확인된 에러 핵심
```text
tar (child): bzip2: Cannot exec: No such file or directory
```

### 원인 판단
- `phantomjs-prebuilt`가 압축 파일(`.tar.bz2`)을 풀 때 `bzip2` 프로그램이 필요함
- 현재 WSL에 `bzip2`가 설치되지 않아 `npm install`이 실패함

### 이 단계의 판단
- 지금은 의존성 구조를 뜯어고치기보다, 먼저 설치를 통과시키는 것이 우선
- 따라서 우선 조치는 `bzip2` 설치 후 `npm install` 재시도

## 7. 다음 세션에서 가장 먼저 실행할 명령
```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y git curl unzip build-essential bzip2
```

그다음:

```bash
curl -fsSL https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install --lts
node -v
npm -v
```

작업 폴더 준비:

```bash
mkdir -p ~/projects
cd ~/projects
git clone <repo-url>
cd company_search_app_web
```

그다음 설치 재시도:

```bash
npm install
```

설치가 끝나면:

```bash
npm run build
npm run preview -- --host 0.0.0.0
```

## 8. 접속 확인 시 주의할 점
- WSL 서버를 띄웠다고 해서 다른 PC가 `WSL 내부 IP`로 바로 접근하는 것은 아님
- 같은 사내망 다른 PC에서는 보통 `윈도우 호스트 IP:포트`로 접근해야 함
- 필요 시 아래를 추가로 맞춰야 함
  - 윈도우 방화벽 포트 허용
  - `netsh interface portproxy` 설정

## 9. Codex 사용 관련 판단
- 앞으로 실제 운영 환경 기준으로 확인해야 할 일이 많으므로, 가능하면 `노트북 WSL 안에 Codex를 설치해서` 그쪽에서 직접 세션을 이어가는 것이 더 정확함
- 이유:
  - 실제 설치/빌드/포트/방화벽 상태를 그 자리에서 바로 확인 가능
  - 개발 PC 환경과 노트북 환경 차이로 인한 혼선을 줄일 수 있음
