# 로그인/계정 운영 방안 정리 (2026-04-02)

## 배경
- 현재 웹서버는 인증(ID/PW) 없이 주소만 알면 누구나 사용 가능.
- 보안 강화를 위해 ID/PW 기반 인증 도입 필요.
- 현재 RDBMS는 사용하지 않음.
- 운영 방식상 회원가입은 필요 없고, 관리자가 계정을 발급하는 방식이 적합.

## 결론
1. JWT보다 서버 세션 기반 인증을 우선 도입한다.
2. 회원가입 기능/회원가입 UI는 만들지 않는다.
3. 관리자가 CLI 스크립트로 계정을 생성/관리한다.
4. 사용자 UI는 최소 2개만 제공한다.
- 로그인 페이지
- 최초 로그인 시 비밀번호 변경 페이지(강제 변경)

## 인증 방식
- 로그인 시 ID/PW 검증 후 서버 세션 생성.
- 브라우저에는 세션 ID 쿠키만 저장.
- 쿠키 보안 옵션 적용:
  - `HttpOnly`
  - `Secure` (HTTPS 환경)
  - `SameSite=Lax` 또는 `SameSite=Strict`
- 보호 라우트는 세션 유효성 검사로 접근 제어.

## 계정 저장 방식 (RDBMS 미사용)
- 파일 기반 저장(`users.json` 또는 `users.yaml`) 사용.
- 비밀번호 평문 저장 금지.
- `argon2id` 또는 `bcrypt` 해시만 저장.
- 권장 필드 예시:

```json
[
  {
    "id": "alice",
    "passwordHash": "$argon2id$...",
    "role": "user",
    "mustChangePassword": true,
    "active": true
  }
]
```

## 관리자 계정 발급/관리 절차
- 회원가입 대신 관리자 전용 스크립트로 계정 생성.
- 예시:
  - `node scripts/create-user.js --id alice`
  - `node scripts/reset-password.js --id alice`
  - `node scripts/disable-user.js --id alice`
- 사용자에게는 `아이디 + 임시 비밀번호` 전달.
- 첫 로그인 시 비밀번호 변경 강제.

## 운영/보안 체크리스트
- HTTPS 강제.
- 로그인 실패 횟수 제한(rate limit).
- 로그인 성공/실패, 계정 생성/비밀번호 변경 감사 로그 기록.
- 세션 저장소 운영 고려:
  - 기본 메모리 스토어는 재시작 시 세션 유실.
  - 운영 환경은 Redis 등 외부 세션 스토어 권장.
- `users.json` 파일 접근권한 최소화 및 백업 암호화.

## 추후 구현 범위(합의)
- 회원가입 API/화면 미구현.
- 로그인/로그아웃/세션체크/비밀번호변경만 구현.
- 관리자용 계정 관리 CLI 스크립트 구현.
