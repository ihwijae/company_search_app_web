# 실적관리 웹 전환 정리

## 1. 목적

이 문서는 기존 GUI(Electron) 기반 실적관리 기능을 웹 전용 구조로 전환하면서 정리된 현재 상태를 기록하기 위한 문서다.

핵심 방향은 다음과 같다.

- 이 저장소의 실적관리는 더 이상 Electron/IPC/SQLite 운영 저장소를 기준으로 하지 않는다.
- 기존 GUI 데이터는 `마이그레이션 입력 포맷`으로만 본다.
- 웹 운영 저장소는 `JSON 메타 + Blob 첨부` 구조를 사용한다.

## 2. 기존 GUI 구조

기존 GUI 실적관리 내보내기 결과는 아래 구조였다.

- `records.sqlite`
- `attachments/{projectId}/...`

즉, 메타 데이터는 SQLite에 있고 실제 PDF/이미지 첨부파일은 별도 폴더에 있었다.

### SQLite에 들어 있던 핵심 정보

- `companies`
  - 법인 정보
- `categories`
  - 공사종류 정보
- `projects`
  - 실적 본문 정보
- `project_categories`
  - 실적과 공사종류 연결 정보
- `attachments`
  - 첨부파일 메타 정보

중요한 점:

- 첨부파일 본문은 SQLite 안에 들어 있지 않았다.
- SQLite에는 파일 경로, MIME, 크기 같은 메타만 있었다.

## 3. 현재 웹 구조

현재 웹 실적관리는 다음 구조를 사용한다.

- 메타 저장소: `company-search/records/index.json`
- 첨부 저장소: `company-search/records/attachments/...`

즉 웹에서는:

- 실적 목록/검색/법인/공사종류/실적 메타는 JSON 기준
- 실제 PDF/이미지 파일은 Blob 기준

### JSON에 들어가는 정보

- companies
- categories
- projects
- 각 project의 attachment 메타
  - `id`
  - `displayName`
  - `fileName`
  - `mimeType`
  - `fileSize`
  - `uploadedAt`
  - `blobPathname`
  - `url`
  - `downloadUrl`

### 첨부 열기 방식

Blob 첨부는 `private` 저장소를 사용한다.

따라서 브라우저가 Blob URL을 직접 여는 방식은 403이 발생할 수 있다.

현재는 아래 경로를 통해 서버가 private Blob를 읽어서 전달한다.

- `/api/records?action=file&pathname=...`

관련 파일:

- [api/records/index.js](/home/leehwijae/projects/company_search_app_web/api/records/index.js)
- [src/shared/recordsWebStore.js](/home/leehwijae/projects/company_search_app_web/src/shared/recordsWebStore.js)

## 4. 현재 운영 기준

실적관리 기능은 이제 Electron fallback 없이 웹 전용으로 동작한다.

관련 원칙:

- 실적 기능은 `window.electronAPI.records`를 사용하지 않는다.
- preload/main의 실적 전용 IPC 연결은 제거했다.
- 실적 저장/조회는 웹 API와 Blob 기준으로 동작한다.

## 5. 마이그레이션 방식

### 왜 브라우저 가져오기 대신 스크립트로 처리했는가

초기에는 웹 화면의 `DB 가져오기`로 기존 GUI 내보내기 폴더를 읽어 마이그레이션하려고 했다.

하지만 실제로는 아래 문제가 있었다.

- 브라우저 폴더 선택 API 제약
- 대량 첨부 업로드 시 디버깅 어려움
- 일부 첨부 실패 시 진행상태 추적이 불편함
- 503/실패 원인 파악이 어려움

그래서 최종적으로는 로컬 스크립트 방식으로 마이그레이션했다.

### 사용한 스크립트

- [scripts/migrate-records-export.js](/home/leehwijae/projects/company_search_app_web/scripts/migrate-records-export.js)
- [scripts/repair-records-attachments.js](/home/leehwijae/projects/company_search_app_web/scripts/repair-records-attachments.js)

### 스크립트 역할

이 스크립트는 기존 GUI 내보내기 폴더를 읽어서:

1. `records.sqlite`를 읽음
2. 회사/공사종류/실적/첨부 메타를 추출
3. `attachments/...` 폴더에서 실제 PDF/이미지 파일을 찾음
4. 첨부파일을 Blob에 업로드
5. 최종 `records/index.json`을 Blob에 저장

### 스크립트 보관 원칙

이 두 스크립트는 일반 사용자 기능이 아니라 관리자/비상 대응용 도구로 유지한다.

- 평소 웹 운영에서는 실행하지 않는다.
- 기존 GUI 내보내기 데이터를 다시 옮겨야 할 때만 사용한다.
- 첨부 누락, 중간 종료, Blob 메타 불일치 같은 장애가 생겼을 때 복구용으로 사용한다.
- Git 이력만 믿고 삭제하지 말고 저장소 안에 유지한다.

현재 기준으로는 마이그레이션과 복구가 끝난 상태이며, 상시 실행 중인 스크립트는 없어야 한다.

### 실행 예시

미리보기만 생성:

```bash
npm run migrate:records -- --dry-run
```

실제 업로드 포함 실행:

```bash
BLOB_READ_WRITE_TOKEN=... npm run migrate:records
```

누락 첨부 복구:

```bash
BLOB_READ_WRITE_TOKEN=... npm run repair:records
```

마이그레이션 당시 확인된 기준 데이터:

- 회사 `8`
- 공사종류 `21`
- 실적 `162`
- 첨부 `183`

기존 `실적관리` 폴더는 분석/마이그레이션 입력용 샘플이었고, 실제 반영 후에는 프로젝트 안에 유지할 필요가 없다.

## 7. 실제 발생했던 문제와 복구

초기 마이그레이션에서는 첨부가 많은 상태에서 업로드 시간이 길어졌고, 스크립트가 마지막에 한 번만 `records/index.json`을 저장하는 구조였다.

그 결과:

- Blob 안에는 일부 첨부 파일이 올라가 있었지만
- `records/index.json`에는 뒤쪽 첨부 메타가 기록되지 못했다
- 실적 화면에서는 첨부가 빠진 것처럼 보였다

실제 확인 시 상태는 아래와 같았다.

- 원본 SQLite 기준 첨부 `183`
- Blob 문서 기준 첨부 `130`
- 누락 구간은 주로 `projectId 111 ~ 163`

즉 원인은 개별 PDF 손상이 아니라:

- 마이그레이션이 중간까지 진행된 뒤
- 마지막 JSON 저장 전/도중에 완료되지 못했던 것

### 복구 방법

이를 해결하기 위해 [scripts/repair-records-attachments.js](/home/leehwijae/projects/company_search_app_web/scripts/repair-records-attachments.js) 를 추가했다.

복구 스크립트 특징:

- 누락된 첨부만 찾는다
- 프로젝트 단위로 Blob 업로드와 JSON 갱신을 함께 처리한다
- 중간에 끊겨도 다음 실행에서 이어서 복구 가능하다
- 전체 재마이그레이션보다 안전하다

복구 후 최종 검증 결과:

- 원본 SQLite 첨부 `183`
- Blob 문서 첨부 `183`
- 차이 있는 프로젝트 수 `0`

즉 현재는 원본 `실적관리` 폴더 기준으로 첨부까지 전부 맞는 상태다.

## 8. 중복/재마이그레이션 동작

재실행 시 같은 첨부 경로는 덮어쓰도록 설계했다.

첨부 업로드 경로:

- `company-search/records/attachments/{projectId}/{attachmentId}-{fileName}`

업로드 옵션:

- `allowOverwrite: true`

즉:

- 같은 project/attachment/fileName이면 새 내용으로 덮어쓴다.
- 새 실적/새 첨부는 추가된다.

실제 복구 과정에서는 `project 164`에 첨부가 한 번 더 들어간 중복 케이스가 있었고, 이후 JSON에서 중복 항목을 제거해 최종 수량을 맞췄다.

## 9. Vercel Blob 사용 시 주의점

Vercel Blob의 고급 작업 한도는 `put()`, `copy()`, `list()` 사용량에 영향을 받는다.

실적관리 기준으로 정리하면:

- `put()`
  - 첨부 업로드
  - `records/index.json` 저장
- `copy()`
  - 현재 실적관리에서는 사용하지 않음
- `list()`
  - Blob 안 파일 목록 스캔 시 발생
  - 실적 화면 진입 자체에서는 사용하지 않음

중요:

- 실적 화면은 첨부 파일명을 Blob `list()`로 읽지 않는다
- 화면은 `records/index.json` 메타만 읽어서 실적과 첨부 파일명을 그린다
- 실제 PDF/이미지 클릭 시에는 특정 파일 하나만 읽는다
- 따라서 실적 화면 자체가 `list()`를 계속 증가시키는 구조는 아니다

한도 절약 원칙:

- 마이그레이션/복구 스크립트를 반복 실행하지 않기
- Blob 대시보드에서 파일 목록 새로고침을 최소화하기
- 대량 재업로드보다 증분 복구를 우선하기

## 10. Vercel 배포 이슈 정리

Hobby 플랜에서는 함수 개수 제한이 있어 기존 API 파일 수가 초과되었다.

그래서 다음 라우트를 통합했다.

- `api/datasets/{status,snapshot,upload}.js` -> `api/datasets/index.js`
- `api/mail/{send-test,send-batch}.js` -> `api/mail/index.js`

결과적으로 실제 함수 수를 12 이하로 줄였다.

## 11. 이후 참고 사항

### 현재 `DB 가져오기`의 의미

지금 실적 화면의 `DB 가져오기`는 일반 운영 기능이라기보다
`기존 GUI 데이터 마이그레이션용` 의미가 강하다.

웹 운영 구조에서는 원칙적으로:

- 모두가 같은 서버 데이터를 공유
- 예전처럼 SQLite 파일을 주고받을 필요가 적음

따라서 나중에는 버튼 문구를 아래처럼 바꾸는 것이 더 맞다.

- `기존 데이터 가져오기`
- `GUI 데이터 이전`

### 현재 `DB 내보내기`의 의미

현재 웹 구조에서는 과거 GUI처럼 SQLite 내보내기 의미가 아니다.

필요하다면 향후:

- 웹 백업 export
- 또는 GUI 복귀용 export

중 하나로 재정의해야 한다.

## 12. 나중에 다시 GUI로 돌아가야 할 때

가능하다. 다만 지금 구조에서 바로 되는 것은 아니고, `역마이그레이션`이 필요하다.

필요한 작업:

1. `records/index.json` 읽기
2. JSON을 SQLite 구조로 변환
3. Blob 첨부를 다운로드
4. `attachments/{projectId}/...` 구조로 복원
5. 최종적으로 아래 묶음 생성
   - `records.sqlite`
   - `attachments/`

즉 개념적으로는 아래 변환 스크립트 하나가 더 있으면 된다.

- `웹 JSON + Blob -> GUI SQLite + attachments`

이 작업은 어렵지 않은 편이다.

이유:

- SQLite 스키마를 이미 알고 있음
- JSON에 동일 정보가 대부분 들어 있음
- 첨부도 Blob 경로를 기준으로 다시 내려받으면 됨
