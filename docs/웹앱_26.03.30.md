## 3) 기존 GUI 구조와 현재 웹 구조
### 기존 GUI
- `records.sqlite`
- `attachments/{projectId}/...`

즉 메타는 SQLite, 실제 첨부는 폴더 저장 구조였다.

## 8) 첨부 열기 방식
- Blob 첨부는 `private` 저장소 사용
- 브라우저가 Blob URL을 직접 열면 `403`이 날 수 있음
- 현재는 `/api/records?action=file&pathname=...` 경로를 통해 서버가 private Blob를 읽어서 전달

즉:
- 화면에서 첨부 파일명을 보여줄 때 `list()`를 하지 않음
- PDF/이미지를 클릭할 때도 `list()`가 아니라 특정 파일 1개를 읽는 구조

## 9) Vercel 함수 수 제한 대응
Hobby 플랜에서 함수 수 제한(`12`)에 걸려 배포가 실패했음.

## 10) 현재 가장 큰 문제: Vercel Blob Advanced Operations
현재 실적관리 구조에서 가장 큰 운영 리스크는 Blob 고급 작업 한도다.

상황:
- Vercel 대시보드 기준 `Advanced Operations 2.1k / 2k`
- 실적 저장/삭제/순서 변경은 `records/index.json` 전체를 다시 저장하므로 `put()` 사용량이 계속 늘 수 있음
- 현재는 동작하더라도 이후 갑자기 Blob 접근이 막힐 가능성을 배제할 수 없음

실적관리 기준 의미:
- `put()`
  - 첨부 업로드
  - `records/index.json` 저장
- `copy()`
  - 현재 실적관리에서 사용 안 함
- `list()`
  - Blob 파일 목록 스캔 때만 사용
  - 실적 화면 진입, 첨부 파일명 표시, PDF 클릭 자체는 `list()`를 반복하지 않음

핵심 판단:
- 현재 병목은 첨부 자체보다 `records/index.json` 전체 저장 구조
- 작은 공사종류 수정도 큰 JSON 전체 저장을 유발함

## 11) 저장소 변경 방향
다음 세션 기준으로 실적관리 저장소는 `Cloudflare R2`로 옮길 계획이다.

현재 판단:
- Vercel Blob 무료 한도는 운영 리스크가 큼
- 첨부 파일 위주 저장소로는 `Cloudflare R2`가 더 유리함
- 다만 메타 저장 구조는 같이 재검토가 필요함

### 다음 세션에서 검토할 방향
1. `Vercel Blob -> Cloudflare R2` 이전 스크립트 작성
2. 현재 `records/index.json` 구조를 그대로 유지할지 검토
3. 가능하면 메타 JSON과 첨부를 더 분리해 작은 변경에 전체 저장이 일어나지 않도록 개선

## 12) R2 전환 시 참고할 점
이전은 가능하다.

이유:
- 현재 Blob 안에 `records/index.json`이 존재
- 각 첨부는 `blobPathname`으로 연결돼 있음
- private Blob도 서버 토큰으로 읽을 수 있음

즉 전환 스크립트는 아래 흐름으로 작성 가능:
1. Blob에서 `records/index.json` 읽기
2. 메타를 새 저장 형식으로 변환
3. 각 첨부를 현재 Blob에서 읽기
4. Cloudflare R2로 업로드
5. 새 저장 경로에 맞게 메타 갱신

즉 `Blob -> 다른 저장소` 마이그레이션 스크립트는 충분히 구현 가능하다.

## 13) GUI로 되돌려야 할 경우
현재 구조는 GUI와 저장 형식이 다르므로 바로 GUI가 읽을 수는 없다.

다만 역방향 변환은 가능하다.

필요한 흐름:
1. 웹 메타 JSON 읽기
2. 다시 SQLite 스키마로 변환
3. Blob 첨부를 내려받아 `attachments/` 폴더 구조 복원
4. 최종적으로 `records.sqlite + attachments/` 패키지 생성

즉 나중에 `웹 -> GUI export` 스크립트를 만들면 다시 GUI로 옮길 수 있다.
