# 협정보드 계산 로직 리팩토링 메모

## 목적

협정보드의 계산 관련 코드가 현재 `AgreementBoardWindow.jsx` 안에 많이 들어 있다.

지금은 기능이 동작하지만, 아래 문제가 있다.

- 계산 코드와 UI 코드가 한 파일에 섞여 있음
- 후보/협정테이블/엑셀 내보내기에서 비슷한 계산 규칙이 반복됨
- 수정 시 영향 범위를 파악하기 어려움
- 선언 순서 및 상태 참조 문제로 버그가 발생하기 쉬움

이 문서의 목적은 `계산 로직을 점진적으로 별도 영역으로 분리하는 방향`을 정리하는 것이다.

## 중요한 전제

### 계산은 하나의 큰 범주로 본다

이번 리팩토링에서는 계산을 억지로 완전히 다른 개념으로 쪼개지 않는다.

즉 아래는 모두 넓게 보면 `계산 로직`이다.

- 경영점수 계산
- 실적 계산
- 총점 계산
- 가능지분 계산
- 경영점수 경고 여부 판단
- 화면에 보여줄 표시값 조립

다만 리팩토링 순서는 위험도를 기준으로 나눈다.

### UI와 계산은 분리한다

목표는 `계산 주체를 없애는 것`이 아니라, `계산 코드를 UI 파일 밖으로 빼는 것`이다.

즉 최종적으로도 협정보드는 계산 결과를 사용하지만,
계산 구현은 별도 파일에서 담당하게 만든다.

## 현재 이해해야 할 구조

현재는 아래 구조다.

- `AgreementBoardWindow.jsx`
  - 계산 코드 + UI 렌더링 + 상태 처리 + 후보 로직 + export payload 일부

- `AgreementCandidateWindow.jsx`
  - 후보창 UI만 담당

- `exportExcel.js`
  - 실제 엑셀 작성 담당

중요한 점

- 후보창은 계산을 직접 하지 않고, 메인에서 계산된 값을 받아서 표시하는 구조에 가깝다.
- 따라서 계산 로직을 분리하더라도 후보창으로 계산 책임을 넘기는 것이 아니다.

## 계산 로직을 어떻게 나눌 것인가

상위 개념은 `계산 로직` 하나로 유지한다.

예상 폴더 예시

- `src/shared/agreements/calculations/`

그 아래에서 세부적으로 분리한다.

예시

- `possibleShare.js`
- `candidateSelectors.js`
- `display.js`
- `score.js`
- `summary.js`

이 구조의 장점

- 계산은 계산끼리 모인다.
- 수정할 때 "이 계산이 어디 있지?"를 찾기 쉽다.
- UI 파일이 계산 구현을 덜 알게 된다.

## 리팩토링 순서

### 1단계: 가능지분 계산 분리

가장 먼저 분리할 대상은 `가능지분 계산`이다.

이유

- 상대적으로 독립적이다.
- 후보/협정테이블/엑셀 이름 셀에서 공통으로 사용한다.
- 점수 산식 핵심보다 위험도가 낮다.

이 단계에서 옮길 것

- 가능지분 계산 함수
- 가능지분 텍스트 포맷팅 함수
- 필요하면 가능지분 표시 여부 판단 함수

예상 파일

- `src/shared/agreements/calculations/possibleShare.js`

### 2단계: 후보 표시용 계산 분리

다음은 후보창에 보여줄 데이터를 만드는 계산을 분리한다.

이 단계에서 옮길 것

- 후보 entries 생성
- 후보 경영 경고 여부
- 후보 가능지분 문자열
- 후보 검색용 텍스트
- 후보 이름/담당자/지역 표시용 값 조립

예상 파일

- `src/shared/agreements/calculations/candidateSelectors.js`

### 3단계: 표시용 공통 계산 분리

협정테이블/후보/엑셀 이름 셀에 공통으로 들어가는 표시 규칙을 정리한다.

이 단계에서 옮길 것

- 업체명 표시 문자열
- 담당자 결합 규칙
- 가능지분 문자열 결합 규칙
- 경영 경고 표시 판단

예상 파일

- `src/shared/agreements/calculations/display.js`

### 4단계: 그룹 요약 계산 분리

그룹별 summary 계산은 규모가 크므로 별도로 분리한다.

이 단계에서 옮길 것

- 그룹별 경영/실적/총점 summary 계산
- 총점 관련 보조 계산

예상 파일

- `src/shared/agreements/calculations/summary.js`

### 5단계: 점수 산식 핵심 분리

가장 마지막에 실제 핵심 점수 산식을 분리한다.

이 단계에서 옮길 것

- 경영점수 산식
- 실적점수 산식
- 발주처/금액대별 분기 계산

예상 파일

- `src/shared/agreements/calculations/score.js`

주의

- 이 단계가 가장 위험하다.
- 앞 단계들로 구조를 정리한 뒤 들어가는 것이 맞다.

## 왜 가능지분부터 시작하는가

가능지분은 계산이긴 하지만, 현재 역할상 다음 특징이 있다.

- 화면 표시와 연결이 강함
- 후보/협정테이블/엑셀 이름 셀에 재사용됨
- 점수 총합이나 배치 로직과 직접 결합도가 상대적으로 낮음

따라서 `계산 로직 분리의 첫 단계`로 적합하다.

## 분리할 때 지켜야 할 원칙

### 1. 한 번에 많이 건드리지 않는다

한 파일에서 여러 계산을 동시에 빼지 않는다.

예시

- 가능지분만 먼저 분리
- 테스트
- 그다음 후보 selector 분리

### 2. UI 구조는 최대한 그대로 둔다

처음에는 `호출 위치는 유지하고 구현만 바깥으로 이동`해야 한다.

### 3. 계산 입력과 출력 형태를 명확히 유지한다

예를 들면

- 입력: `candidate`, `possibleShareBase`
- 출력: `possibleShare`, `possibleShareText`

처럼 함수 계약을 단순하게 유지한다.

### 4. 엑셀용 계산과 화면용 계산은 책임을 분리한다

- 화면에서 어떻게 보일지: 프론트 계산 로직
- 엑셀 셀 색상/셀 값 쓰기: `exportExcel.js`

엑셀 셀 스타일 조건은 계산 파일이 아니라 export 파일이 담당한다.

## 이번 세션 기준 현황

이미 완료된 것

- export payload 조립 분리
  - `agreementExportPayload.js`

다음 시작점

1. 계산 로직 폴더 생성
2. `possibleShare.js` 생성
3. 가능지분 계산/문자열 생성 로직 이동
4. 후보/협정테이블/엑셀 이름 셀에서 그 함수 재사용

## 인계용 요약

다음 작업자는 아래 순서로 진행하면 된다.

1. `src/shared/agreements/calculations/` 폴더 생성
2. `possibleShare.js`부터 분리
3. `candidateSelectors.js` 분리
4. `display.js` 분리
5. 마지막에 `summary.js`, `score.js` 분리

핵심 원칙

- 계산은 하나의 큰 영역으로 본다.
- 다만 위험도가 낮은 계산부터 점진적으로 이동한다.
- UI 구조 변경보다 계산 구현 이동을 우선한다.
