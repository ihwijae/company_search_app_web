# 협정보드 / 지역사찾기 공통화 정리 (2026-03-17)

## 배경

- 지역사찾기와 협정보드가 `단독가능/불가능`을 서로 다른 함수로 판단하고 있었음
- 특히 `행안부 30억~50억` 구간에서
  - `참가자격 없음`이어도 `참가자격 0`으로 비교되는 문제
  - 지역사찾기 실적 만점 기준이 협정보드와 다르게 `기초금액` 쪽으로 흐르는 문제
  - 참가자격 방식 버튼 변경 시 지역사찾기 창이 다시 튀는 문제
  가 확인됨

## 이번에 바꾼 내용

### 1. 단독판정 공통 evaluator 도입

- 공통 단독판정 로직 파일 추가
  - 렌더러용: `src/shared/agreements/rules/singleBidEligibility.js`
  - 메인용: `src/shared/agreements/rules/singleBidEligibility.cjs`
- 공통 evaluator 이름:
  - `evaluateSingleBidEligibility`
- 반환값:
  - `ok`
  - `reasons`
  - `facts`
  - `entry`
  - `performance`
  - `management`
  - `region`

### 2. 협정보드 단독가능 뱃지 공통화

- `AgreementBoardWindow.jsx` 내부 로컬 단독판정 로직이 공통 evaluator를 호출하도록 변경
- 협정보드는 공통 evaluator 결과 중 `ok`만 사용해 뱃지를 표시

### 3. 지역사찾기 단독판정 공통화

- `main.js` 후보 평가 로직이 공통 evaluator를 사용하도록 변경
- `singleBidEligible`, `singleBidReasons`, `singleBidFacts` 외에
  `singleBidDetails`도 함께 내려주도록 변경
- 지역사찾기 UI는 기존 표시를 유지하면서 공통 판정 결과를 사용

### 4. 참가자격 없음 처리 수정

- `entryMode = none`이면 참가자격 비교를 완전히 생략하도록 수정
- 더 이상 `시평 미달: ... < 참가자격 0` 사유가 나오면 안 되는 방향으로 수정

### 5. 행안부 30억~50억 실적 만점 기준 정합

- 협정보드 기준 확인:
  - `행안부 30억~50억`은 `추정가격 × 80%`
- 지역사찾기에도 같은 값을 넘기도록 수정
  - `perfectPerformanceAmount`
  - `perfectPerformanceBasis`
- 메인 후보 평가에서도 `perfectPerformanceAmount`를 우선 사용하도록 수정
- 사유 문구도 `perfectPerformanceBasis`를 사용할 수 있게 반영

### 6. 지역사찾기 팝업 자동 재동기화 완화

- 열린 지역사찾기 창이 폼 변경 때마다 다시 튀거나 포커스를 가져오는 문제를 줄이기 위해
  자동 props 동기화 효과를 제한
- 현재는 다시 열거나 다시 검색할 때 반영하는 방향

## 실적 관리 관련 같이 반영된 내용

### 1. 기본값 재시드 방지

- 실적 관리 공사종류/법인 삭제 후 재실행하면 기본값이 다시 살아나던 문제 수정
- 최초 1회만 기본값을 시드하고 이후에는 사용자 DB 상태를 유지하도록 변경

### 2. 공사종류 순서 변경 버그 수정

- `sort_order` 중복으로 인해 특정 항목 사이에 이동이 막히는 문제 수정
- 이동 시 형제 목록 전체 `sortOrder`를 재번호 매기도록 변경
- 기존 DB도 마이그레이션에서 순서를 재정렬하도록 보정

## 남은 확인 포인트

- 실제 앱에서 수동 검증은 아직 안 함
- 다음 세션에서 우선 확인할 것:
  1. `행안부 30억~50억`에서 지역사찾기 창 완전 종료 후 재오픈
  2. `참가자격 산출 방식 = 없음` 상태에서 `참가자격 0` 사유가 사라졌는지 확인
  3. `행안부 30억~50억`에서 지역사찾기 실적 기준이 `추정가격 × 80%`로 동작하는지 확인
  4. 같은 업체 기준으로 협정보드 `단독가능` 뱃지와 지역사찾기 `단독가능/불가능` 결과가 일치하는지 확인
  5. 참가자격 방식 버튼 클릭 시 지역사찾기 창이 다시 튀지 않는지 확인

## 메모

- 공통 로직은 현재 ESM/CJS 이중 파일로 유지
  - 이유: 렌더러는 `import`, 메인은 `require` 사용
- 장기적으로 메인/렌더러 모듈 체계를 더 통일하면 공통 규칙 파일도 단일 소스로 정리 가능
