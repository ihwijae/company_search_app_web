# 작업 규칙

이 문서는 이 저장소에서 작업할 때 세션 시작 시 가장 먼저 확인하는 기본 규칙이다.

## 1. 모달/알림/팝업 UI 규칙

- 모달창, 알림창, 확인창, 팝업창은 브라우저 기본 `alert`, `confirm`, `prompt` 또는 운영체제 기본 팝업에 의존하지 않는다.
- 가능한 한 앱 내부의 자체 UI를 사용한다.
- 공통 알림/확인 UI가 필요한 경우 `src/components/FeedbackProvider.jsx`와 관련 스타일을 우선 사용한다.
- 새 입력 UI가 필요할 때도 브라우저 기본 팝업보다 앱 내부 패널, 자체 다이얼로그, 별도 화면 등 포커스를 직접 제어할 수 있는 구조를 우선한다.

## 2. 기능 분리/설계 규칙

- `docs` 폴더의 협정보드 리팩토링 관련 문서들에 정리된 방향처럼, 기능 구현 시 로직과 UI를 한 파일에 계속 누적하지 말고 역할별로 분리하는 설계를 우선한다.
- 새로운 기능을 만들거나 기존 기능을 확장할 때도 동일한 원칙을 적용한다.
- 특히 아래 기준을 지킨다.
- 화면 컴포넌트, 상태 관리, 비즈니스 로직, 데이터 처리, 공통 UI를 가능한 한 분리한다.
- 한 파일이 과도하게 커지면 하위 컴포넌트, 훅, 유틸, 서비스 레이어로 나누는 방향을 먼저 검토한다.
- 기존 기능에 임시 코드만 덧붙이는 방식보다, 이후 유지보수와 확장을 고려한 구조로 정리한다.
- 관련 작업 전에는 필요 시 `docs` 폴더의 협정보드 리팩토링 문서를 참고해 현재 저장소의 설계 방향과 맞추도록 한다.

## 3. 플랫폼 기준 규칙

- 이 저장소는 웹 버전만을 대상으로 작업한다.
- 이 프로젝트의 방향은 `기존 GUI(Electron) 앱을 웹으로 완전히 전환`하는 것이다.
- 기존 GUI(Electron) 전용 앱은 별도 프로젝트로 따로 관리한다고 가정한다.
- 따라서 이 저장소에서는 GUI/Electron 쪽 구현을 병행 유지하지 않는다.
- Electron, IPC, preload, 데스크톱 전용 분기 호환을 유지하기 위한 코드는 새로 추가하지 않는다.
- Electron 구현을 유지하기 위한 fallback, 분기, 브리지 재사용 코드도 추가하지 않는다.
- 웹 구현 중 `기존 Electron API가 있으면 그 경로를 우선 사용`하는 식의 임시 호환 코드는 금지한다.
- 기존 데스크톱 전용 코드가 남아 있더라도, 웹 기능으로 대체 가능한 경우 웹 기준으로 정리하고 제거하는 방향을 우선한다.
- 남아 있는 Electron 관련 코드는 이후 단계적으로 삭제하는 방향을 기본 원칙으로 삼는다.
- 구현 시 `웹/데스크톱 겸용`보다 `웹 전용으로 단순화`를 우선한다.
- 같은 기능을 웹과 데스크톱 두 경로로 동시에 유지하는 방식은 특별한 이유가 없는 한 피한다.

## 4. 개발 환경 / 인계 규칙

- 이 저장소의 기본 개발 환경은 `Windows + WSL + VS Code Remote` 기준으로 본다.
- 실제 개발과 명령 실행은 가능하면 `WSL 내부`에서 진행한다.
- 새 PC에서 Codex에 작업을 넘기기 전, 개발자가 먼저 `WSL2`, `Ubuntu`, `Node.js`, `npm`, `git`까지는 설치해 둔 상태여야 한다.
- `Codex CLI` 설치는 `Node.js`와 `npm`이 먼저 있어야 가능하므로, Codex 설치 전 `node -v`, `npm -v`가 WSL에서 정상 동작해야 한다.
- 새 PC 복구나 인계 시에는 루트 환경변수 파일과 데이터 폴더(`.env`, `.env.local`, `db/`, `template/`, `템플릿/`) 복원 여부를 먼저 확인한다.
- WSL 복구 절차와 설치 스크립트는 `docs/pc백업.md`와 `scripts/setup-wsl-dev.sh` 기준으로 진행한다.
